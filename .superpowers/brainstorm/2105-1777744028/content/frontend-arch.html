Frontend Architecture

How should the Next.js app(s) be structured?

A

Single Next.js App

One repo, one deployment. Admin panel, POS, and customer storefront all live under separate route groups: /admin/*, /pos/*, /(shop)/*. Role-based middleware controls access. Shared UI components, one build pipeline.

Pros

  • One codebase to maintain
  • Shared auth + component library
  • Simpler deployment

Cons

  • Bundle size if not code-split carefully
  • Admin and customer code in same repo
B

Two Next.js Apps

Separate apps: admin-app (staff + POS + inventory) and customer-app (storefront + orders). Each deployed independently. Can share a component library as an npm package.

Pros

  • Clean separation of concerns
  • Independent deployments and scaling
  • Different teams can own each app

Cons

  • Two repos / two pipelines to manage
  • Shared code needs an npm package or monorepo
  • More setup overhead

Recommendation

Option A is the pragmatic choice for MVP — one codebase, faster to build, easy to split later if needed. Next.js route groups keep admin and customer routes cleanly separated at zero extra cost.
